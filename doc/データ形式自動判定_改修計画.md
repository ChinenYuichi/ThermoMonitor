# データ形式自動判定 改修計画
## v2.2.0 → v2.3.0

**作成日:** 2026-04-21  
**前提バージョン:** v2.2.0（FC03自動認識実装済み・動作不可確認済み）

---

## 1. 調査結果まとめ

### 1.1 FC03 自動認識が失敗した根本原因

CPSN-MCB271 ソフトウェア編リファレンスマニュアルを確認した結果、**CPSN-SSI-4C は FC 0x03（Read Holding Registers）を実装していない**ことが判明した。

| FC | 機能 | SSI-4C対応 |
|---|---|---|
| 0x03 | Read Holding Registers | **非対応**（例外コード 0x02 を返す） |
| 0x04 | Read Input Registers | 対応（現在使用中） |

Web設定画面（センサタイプ・データ形式）の内容はModbusレジスタに公開されておらず、**Modbusによるポーリングで設定値を読み取ることは不可能**である。

### 1.2 CPSN-SSI-4C の Modbus データ形式（3種類）

Web設定「Modbusデータ形式」により、FC04 で返される 32bit データの構造が根本的に変わる。

#### 形式1: 測定値(0～4294967295) ← **現在の実機設定・動作確認済み**
```
D31 D30 D29 D28 D27 D26 D25 D24 | D23 ～ D0
[センサHF][ADCHF][CJHF][CJソフト][上限][下限][ADC範囲外][有効] | 温度データ(24bit符号付き)
温度 = D23-D0 / 1024  [℃]
```

#### 形式2: 温度(単位:0.1℃)+ステータス
```
+00h: T15～T0  (16bit符号付き整数, 温度 = 値 / 10 [℃])
+01h: 0×8 | F7 F6 F5 F4 F3 F2 F1 F0  (フォルトフラグ)
```

#### 形式3: 温度(単位:0.001℃)
```
+00h: T31～T16  (32bit符号付き整数の上位16bit)
+01h: T15～T0   (32bit符号付き整数の下位16bit)
温度 = int32値 / 1000  [℃]  ※フォルトビットなし
```

### 1.3 形式の違いをビットパターンで区別できるか

FC04 で読んだ生データから形式を推定できる。キーとなるのは **D24ビット（32bit値のビット24）** の挙動である。

| 形式 | 正温度・正常時の D24 | 根拠 |
|---|---|---|
| 測定値 (Raw SSI) | **常に 1**（有効ビット） | 仕様で D24=1 が "データ有効" を示す |
| 0.001℃ | 0（正温度 < 16,777℃の場合） | int32の bit24 は正の小さな値では必ず 0 |
| 0.1℃+ステータス | 0（温度 < 65,536℃の場合） | reg0は温度×10の16bit値、bit8はほぼ 0 |

**判定アルゴリズム（多数決）:**
1. 全4チャンネルの 32bit 生データを取得
2. 各チャンネルで `(data >> 24) & 0xFF` = 上位バイト を取得
3. 上位バイトが `0x01`（D24=1, D25-D31=0）→ Raw SSI 票 +1
4. 上位バイトが `0x00` かつ上位16bit が 0x001C 以下 → 0.001℃ 票 +1
5. 上位バイトが `0x00` かつ下位16bit の上位8bit が 0x00 → 0.1℃+ST 票 +1
6. 最多票の形式を採用。同数なら Raw SSI をデフォルトとする

**信頼性:**

| 状況 | 判定精度 |
|---|---|
| センサ接続・正温度（0℃以上）で正常読取中 | **高い（~95%）** |
| センサ未接続（フォルト状態） | 低い（形式間で特殊値が重複する） |
| 負温度（0℃未満） | やや低い（0.001℃の負値とRaw SSIが混在する可能性） |

## 2. 設計決定

| 項目 | 決定内容 |
|---|---|
| FC03 自動認識 | **廃止**（`Modbus_ReadHoldingRegisters`・`ADDR_*` 定数を削除） |
| 新たな自動判定方式 | FC04 データのビットパターンによるヒューリスティック検出 |
| センサタイプの取得 | Modbus では不可能 → UIR にリングコントロールを追加してユーザーが手動選択 |
| データ形式の解析 | `DataFormat` 列挙型を定義し、形式別に解析ロジックを分岐 |
| 判定失敗時のフォールバック | Raw SSI（現在動作確認済みの設定）をデフォルトとする |

## 3. UIR 改修計画

### 3.1 既存コントロールの役割変更

| コントロール | 現在の役割 | 変更後の役割 |
|---|---|---|
| `PANEL_SENSOR_INFO` (textMsg, ID=49) | FC03 認識結果表示（廃止） | **データ形式 自動判定結果表示** |

表示例（変更後）:
- 判定成功: `データ形式: Raw SSI [自動判定]`
- 判定成功: `データ形式: 0.001℃ [自動判定]`
- 判定失敗: `データ形式: Raw SSI [デフォルト]`

### 3.2 新規追加コントロール

#### ① `PANEL_SENSOR_TYPE_SEL` — センサタイプ手動選択リング

Modbusではセンサタイプを読み取れないため、ユーザーが手動選択する。  
選択値はグラフY軸の推奨レンジ表示に使用する。

| 属性 | 設定値 |
|---|---|
| **コントロール型** | Ring（リング） |
| **Constant Name** | `SENSOR_TYPE_SEL`（CVIが `PANEL_SENSOR_TYPE_SEL` と生成） |
| **ラベル** | `センサタイプ` |
| **選択肢** | `J型` / `K型` / `E型` / `N型` / `R型` / `S型` / `T型`（0〜6） |
| **デフォルト値** | `K型`（index=1） |
| **配置位置** | `PANEL_SENSOR_INFO` の直下 |
| **Callback** | `SensorTypeSel_Callback`（選択変更時に推奨レンジをログ出力） |

#### ② `PANEL_FORMAT_CONFIDENCE` — 判定信頼度表示

自動判定の確信度（何チャンネルが判定に貢献したか）をテキストで表示する。

| 属性 | 設定値 |
|---|---|
| **コントロール型** | textMsg |
| **Constant Name** | `FORMAT_CONFIDENCE` |
| **初期テキスト** | `(判定待機中)` |
| **配置位置** | `PANEL_SENSOR_TYPE_SEL` の直下 |
| **Callback** | なし |

表示例:
- `信頼度: 高 (4/4チャンネル一致)`
- `信頼度: 中 (2/4チャンネル一致)`
- `信頼度: 低 (フォルト多数・デフォルト適用)`

### 3.3 UIR エディタ操作手順

```
Step A: ThermoMonitor.uir をUIRエディタで開く

Step B: PANEL_SENSOR_TYPE_SEL（リングコントロール）を追加
  1. Create → Ring でコントロールを作成
  2. Constant Name: SENSOR_TYPE_SEL
  3. Label: センサタイプ
  4. Items: J型(0) / K型(1) / E型(2) / N型(3) / R型(4) / S型(5) / T型(6)
  5. Default: 1 (K型)
  6. Callback: SensorTypeSel_Callback
  7. PANEL_SENSOR_INFO の直下に配置

Step C: PANEL_FORMAT_CONFIDENCE（textMsg）を追加
  1. Create → Text Message でコントロールを作成
  2. Constant Name: FORMAT_CONFIDENCE
  3. 初期テキスト: (判定待機中)
  4. Callback: なし
  5. PANEL_SENSOR_TYPE_SEL の直下に配置

Step D: 保存して ThermoMonitor.h の自動再生成を確認
  確認する定数名:
  - PANEL_SENSOR_TYPE_SEL または PANEL_PANEL_SENSOR_TYPE_SEL
  - PANEL_FORMAT_CONFIDENCE または PANEL_PANEL_FORMAT_CONFIDENCE
  ※ Constant Name に PANEL_ を付けた場合は二重プレフィックスになる。
     今回は SENSOR_TYPE_SEL / FORMAT_CONFIDENCE と設定（PREFIXなし）
     → CVI は PANEL_SENSOR_TYPE_SEL / PANEL_FORMAT_CONFIDENCE と生成する
```

## 4. コード改修計画

### 4.1 `modbus_tcp.h`

#### 削除
```c
// 削除: FC03 関連（SSI-4C に保持レジスタなし）
#define ADDR_SENSOR_TYPE   0x00
#define ADDR_DATA_FORMAT   0x02
int Modbus_ReadHoldingRegisters(...);
uint8_t Modbus_GetLastException(void);
```

#### 追加
```c
/* データ形式列挙型 */
typedef enum {
    DATA_FORMAT_RAW_SSI  = 0,   /* 測定値(0～4294967295): D24=有効ビット, 温度/1024 */
    DATA_FORMAT_TEMP_01  = 1,   /* 温度(0.1℃)+ステータス: reg0=温度x10, reg1=フォルト */
    DATA_FORMAT_TEMP_001 = 2    /* 温度(0.001℃): int32/1000, フォルトビットなし */
} DataFormat;

/* データ形式名称文字列 */
extern const char *DataFormatNames[3];

/* データ形式ヒューリスティック検出 */
DataFormat SSI4C_DetectDataFormat(int unitId, int startAddr, int *confidence);

/* シグネチャ変更: divisor → DataFormat */
int  SSI4C_ReadAllChannels(int unitId, int startAddr, ChannelData *chData, DataFormat fmt);
void SSI4C_ParseData(uint32_t rawHigh, uint32_t rawLow, ChannelData *chData, DataFormat fmt);
```

> **注意:** `SSI4C_ParseData` の引数を `uint32_t ssiData` 単体から `rawHigh`/`rawLow` の2引数に変更する。  
> 0.1℃+ステータス形式では上位16bit（温度）と下位16bit（フォルト）を独立して扱う必要があるため。

### 4.2 `modbus_tcp.c`

#### 削除
- `Modbus_ReadHoldingRegisters()` 関数全体
- `Modbus_GetLastException()` 関数全体
- `g_lastException` 変数

#### 追加: `SSI4C_DetectDataFormat()`
```
処理フロー:
1. FC04 で 8レジスタ（全4チャンネル）を1回読み取る
2. 各チャンネルの reg0(上位16bit) / reg1(下位16bit) を取得
3. 判定ロジック（チャンネルごと）:
   - (reg0 & 0xFF00) >> 8 == 0x01 かつ (reg0 & 0xFE00) == 0 → Raw SSI 票
   - reg0 <= 0x001C かつ reg0 >= 0x0000 かつ reg1 が有効範囲 → 0.001℃ 票
   - reg0 upper byte が 0x00 かつ reg1 upper byte が 0x00 → 0.1℃+ST 票
4. 最多票の形式を返す。*confidence = 一致チャンネル数 (0-4)
5. 票が同数 or 全チャンネルフォルト → DATA_FORMAT_RAW_SSI を返し *confidence = 0
```

#### 変更: `SSI4C_ReadAllChannels()`
```c
/* 変更前 */
int SSI4C_ReadAllChannels(int unitId, int startAddr, ChannelData *chData, double divisor);

/* 変更後 */
int SSI4C_ReadAllChannels(int unitId, int startAddr, ChannelData *chData, DataFormat fmt);
```
内部で `SSI4C_ParseData(reg[ch*2], reg[ch*2+1], &chData[ch], fmt)` を呼ぶ。

#### 変更: `SSI4C_ParseData()`

```
DataFormat 別解析ロジック:

[DATA_FORMAT_RAW_SSI] ← 現在の実装をほぼ流用
  ssiData = (rawHigh << 16) | rawLow
  faultByte = (ssiData >> 24) & 0xFF
  isValid = (ssiData & FAULT_VALID_BIT) ? 1 : 0
  temperature = SignExtend24(ssiData & 0x00FFFFFF) / 1024.0

[DATA_FORMAT_TEMP_01]
  int16_t tempRaw = (int16_t)rawHigh  ← reg0 が温度データ
  uint8_t faultFlags = (uint8_t)(rawLow & 0xFF)  ← reg1 下位バイトがフォルト
  temperature = tempRaw / 10.0
  isValid / hasHardFault / hasSoftFault をフォルトフラグから解析

[DATA_FORMAT_TEMP_001]
  int32_t tempRaw = (int32_t)((rawHigh << 16) | rawLow)
  temperature = tempRaw / 1000.0
  isValid = 1（この形式にフォルトビットなし）
  hasHardFault = 0 / hasSoftFault = 0
  ※ 特殊値 (-999000 等) でフォルトを示す場合は要確認
```

### 4.5 `ThermoMonitor.c`

#### 内部変数の変更
```c
/* 削除 */
static double g_currentDivisor    = 1024.0;
static int    g_currentSensorType = SENSOR_TYPE_K;

/* 追加 */
static DataFormat g_currentDataFormat = DATA_FORMAT_RAW_SSI;   /* デフォルト: Raw SSI */

```

#### `AutoRecognizeDeviceSettings()` → `DetectAndApplyDataFormat()` に置き換え
```
処理フロー:
1. SSI4C_DetectDataFormat(unitId, 0, &confidence) を呼ぶ
2. g_currentDataFormat = 判定結果
3. g_formatConfidence = confidence
4. PANEL_SENSOR_INFO に判定結果を表示
   例: "データ形式: Raw SSI [自動判定]"
5. PANEL_FORMAT_CONFIDENCE に信頼度を表示
   例: "信頼度: 高 (4/4チャンネル一致)"
6. ログに判定結果・推奨グラフレンジを出力
```

#### `SensorTypeSel_Callback()` を追加
```
EVENT_COMMIT 時:
1. ring コントロールからセンサタイプインデックスを取得
2. センサタイプ別の推奨Y軸レンジをログに出力
   例: "センサタイプ変更: K型 / 推奨グラフレンジ: -100〜1400℃"
```

#### `ReadAndDisplayTemperatures()` の変更
```c
/* 変更前 */

/* 変更後 */

```

#### 切断時のリセット処理（各所）
```c

ATTR_CTRL_VAL → "(判定待機中)"
/* PANEL_SENSOR_TYPE_SEL はリセット不要（ユーザー設定を維持） */
```

## 8. 制約・注意事項

| 項目 | 内容 |
|---|---|
| 判定精度の限界 | センサ全チャンネルがフォルト状態の場合、形式判定が不安定になる。フォルト時はデフォルト（Raw SSI）を維持する |
| 0.1℃+ステータス形式のフォルトフラグ | マニュアルに F0-F7 の詳細定義が不明確なため、実機確認が必要 |
| 0.001℃形式のフォルト表現 | フォルト時の特殊値（-999000 等の可能性）は実機確認が必要 |
| センサタイプの手動選択 | Modbusでは読み取り不可。UIRのリングで設定した値はセッション間で保持されない（将来的に設定ファイル保存を検討） |
| FC03 関連コードの完全削除 | v2.2.0 で追加した FC03 コードを漏れなく削除すること |
```
